## 摘要

本白皮书阐述"基于AI规则引擎的内部存货交易未实现利润自动抵消系统"的技术架构、核心算法、规则引擎设计及可扩展性规划。系统以Python生态为基础，采用JSON可配置规则引擎替代传统硬编码业务逻辑，实现顺流/逆流/平流三类内部交易未实现利润的自动识别、计算与抵消分录生成，并通过本地哈希日志保证计算过程可追溯。

### 1.3 为什么不用Java企业级架构？

本系统V1.0定位于**原型验证**而非生产系统。选择Python+Streamlit：

1. **开发效率**：1个Python文件（~500行）即可完成全部功能，Java企业级架构需要至少5个子项目
2. **演示效果**：Streamlit的Web界面可直接演示，无需额外部署
3. **评审友好**：老师可以直接运行并查看结果，无需配置数据库/中间件
4. **核心逻辑等价**：抵消计算的核心算法（规则引擎+分录生成）与Java实现完全一致，架构思想可平移

## 2. 核心算法设计

### 2.1 交易类型识别算法

**问题**：给定销售方A和购买方B，判断交易类型。

**算法**：基于股权关系图谱的最近公共祖先（LCA）判定。

```
Algorithm: classify_trade(A, B, equity_graph)

Input:
  A: 销售方主体编码
  B: 购买方主体编码
  equity_graph: 股权关系图 {parent: {child: ratio}}

Output:
  trade_type ∈ {DOWN, UP, FLAT, UNKNOWN}
  minority_ratio ∈ [0, 1]  (仅UP和FLAT需要)

1. 若 equity_graph.get_parent(B) == A:
     return (DOWN, None)       // A是B的母公司 → 顺流

2. 若 equity_graph.get_parent(A) == B:
     ratio = equity_graph[B][A]
     return (UP, 1 - ratio)    // B是A的母公司 → 逆流

3. parent_A = equity_graph.get_parent(A)
   parent_B = equity_graph.get_parent(B)
   若 parent_A == parent_B 且 parent_A 不为空:
     ratio = equity_graph[parent_A][A]
     return (FLAT, 1 - ratio)  // 同一母公司 → 平流

4. return (UNKNOWN, None)      // 无法判定
```

**复杂度**：O(1)，股权关系预加载为哈希表。

### 2.2 未实现利润计算算法

**问题**：已知内部销售收入R、成本C、销售数量Q、期末结存数量S，计算未实现利润U。

```
Algorithm: calculate_unrealized_profit(R, C, Q, S, method)

Input:
  R: 销售收入（不含税）
  C: 销售成本
  Q: 内部销售总数量
  S: 期末结存数量
  method: 存货计价方法 ∈ {SPECIFIC_ID, FIFO, WEIGHTED_AVG}

Output:
  U: 未实现利润

  gross_margin_rate = (R - C) / R
  unsold_ratio = S / Q

  case method of:
    SPECIFIC_ID:
      U = R × gross_margin_rate × unsold_ratio
    FIFO:
      // 假设先购入的先售出，期末库存为后购入批次
      // V1.0在Excel模板层由用户按批次分别填写
      // 故单批次内的FIFO等价于SPECIFIC_ID
      U = R × gross_margin_rate × unsold_ratio
    WEIGHTED_AVG:
      // V2.0+支持，V1.0不实现
      pass

  return U
```

### 2.3 抵消分录生成算法

**问题**：根据交易类型T、未实现利润U、少数股东比例M，生成抵消分录列表。

**算法**：查表法（Lookup Table），分录模板预配置在JSON中。

```
Algorithm: generate_entries(T, R, U, M, tax_rate)

Input:
  T: 交易类型 ∈ {DOWN, UP, FLAT}
  R: 内部销售收入
  U: 未实现利润
  M: 少数股东比例（UP/FLAT场景，DOWN为None）
  tax_rate: 所得税税率（默认0.25）

Output:
  entries: [(dr_cr, account, amount), ...]

  template = RULE_CONFIG["entry_templates"][T]
  entries = []

  // 遍历模板中的每条分录规则
  for each rule in template["entries"]:
    amount = eval(rule["formula"])  // 在安全上下文中计算
    app(entries, (rule["dr"], rule["cr"], amount))

  // 追加少数股东分录
  for each rule in template["minority_entries"]:
    amount = eval(rule["formula"])
    app(entries, (rule["dr"], rule["cr"], amount))

  return entries
```

**安全上下文**：`eval` 仅在受限命名空间执行，可用的变量仅限于 `{R, U, M, tax_rate, unsold_ratio}`，不允许访问系统函数。

## 3. AI规则引擎设计

### 3.1 设计哲学

传统做法将抵消规则硬编码在程序逻辑中：

```python
# ❌ 硬编码方式
if trade_type == "DOWN":
    entries.append(("借", "营业收入", revenue * unsold_ratio))
    entries.append(("贷", "存货", unrealized_profit))
```

本系统采用**可配置规则引擎**：抵消模板存储在JSON配置文件中，核心引擎读取配置→匹配规则→执行计算。

```python
# ✅ 规则引擎方式
template = rule_config["entry_templates"][trade_type]
for rule in template["entries"]:
    amount = safe_eval(rule["formula"], context)
    entries.append((rule["dr"], rule["cr"], amount))
```

### 3.2 规则配置文件结构

```json
{
  "version": "1.0",
  "tax_rate": 0.25,
  "cost_methods": ["SPECIFIC_ID", "FIFO"],
  "entry_templates": {
    "DOWN": {
      "label": "顺流交易（母→子）",
      "description": "未实现利润全部归属于母公司股东",
      "entries": [
        {
          "dr": "营业收入",
          "cr": "营业成本",
          "formula": "revenue * unsold_ratio - unrealized_profit",
          "note": "倒挤营业成本"
        },
        {
          "dr": null,
          "cr": "存货",
          "formula": "unrealized_profit",
          "note": "冲减存货至原销售方成本"
        },
        {
          "dr": "递延所得税资产",
          "cr": "所得税费用",
          "formula": "unrealized_profit * tax_rate",
          "note": "确认递延所得税资产"
        }
      ],
      "minority_entries": []
    }
  }
}
```

### 3.3 规则引擎的优势

| 特性 | 硬编码方式 | 规则引擎方式 |
|------|:--------:|:----------:|
| 新增抵消场景 | 修改代码+测试+重新部署 | 修改JSON配置，无需改代码 |
| 多会计准则切换 | 维护多套代码分支 | 切换JSON配置文件 |
| 非技术人员调整 | 需开发介入 | 会计人员可直接修改规则文件 |
| 规则可审计 | 分散在代码各处 | 集中在一个JSON文件中 |
| 课程展示价值 | 低 | **高——体现了"AI规则驱动"的设计思想** |

### 3.4 为什么叫"AI规则引擎"？

本系统的"AI"体现在三个层面：

1. **智能识别**：系统根据股权关系图谱自动判定交易类型，替代人工逐笔翻阅股权结构表
2. **规则驱动**：抵消逻辑以声明式规则（declarative rules）表达，系统根据规则自动推导分录，而非if-else堆砌
3. **可学习性**：JSON规则文件可被AI大模型读取和理解。未来扩展中，可通过自然语言指示（"新增固定资产内部交易抵消规则"）让AI自动生成对应的JSON规则条目

## 4. 数据流设计

### 4.1 数据流转全链路

```
Excel文件上传
  │
  ├─ 表A：内部销售明细 → pandas DataFrame
  ├─ 表B：存货期末结存 → pandas DataFrame
  └─ 表C：股权关系配置 → pandas DataFrame
        │
        ▼
    数据校验器（Validator）
    ├── 列名完整性检查
    ├── 数据类型检查（数值列非字符串）
    └── 交叉引用检查（批次号、主体编码一致性）
        │
        ▼
    数据合并（merge）
    以 批次号 + 主体编码 为键，合并表A和表B
        │
        ▼
    类型识别器（Classifier）
    对合并后的每一行，调用 classify_trade()
    新增列：trade_type, minority_ratio
        │
        ▼
    利润计算器（Calculator）
    对每一行，调用 calculate_unrealized_profit()
    新增列：gross_margin_rate, unsold_ratio, unrealized_profit
        │
        ▼
    分录生成器（EntryGenerator）
    对每一行，调用 generate_entries()
    输出：消除分录列表（展开为多行）
        │
        ▼
    日志记录器（Logger）
    每笔分录写入 SQLite elim_entry 表
    同时生成 JSON 计算日志 + SHA256哈希
        │
        ▼
    结果输出
    ├── Streamlit Web展示（DataFrame + 会计分录格式）
    ├── Excel导出（openpyxl写入）
    └── PDF导出（ReportLab生成）
```

### 4.2 关键数据结构（内存中）

```python
@dataclass
class TradeRecord:
    """单笔内部交易"""
    batch_no: str
    seller_code: str
    buyer_code: str
    item_name: str
    revenue: float
    cost: float
    quantity: int
    period: str
    # 以下为计算过程中追加的字段
    trade_type: Optional[str] = None        # DOWN/UP/FLAT/UNKNOWN
    minority_ratio: Optional[float] = None
    gross_margin_rate: Optional[float] = None
    unsold_ratio: Optional[float] = None
    unrealized_profit: Optional[float] = None

@dataclass
class EliminationEntry:
    """单笔抵消分录行"""
    elim_no: str
    trade: TradeRecord
    dr_cr: str                             # D/C
    account_name: str
    amount: float
    calc_params: dict                       # 计算参数快照
    log_hash: str                           # SHA256
```

## 5. 可追溯性设计

### 5.1 计算日志

每笔抵消分录生成时，系统同步输出JSON格式的计算日志：

```json
{
  "elim_no": "ELIM-202605-0003",
  "timestamp": "2026-05-14T15:30:00+08:00",
  "input": {
    "seller": "P001",
    "buyer": "S001",
    "revenue": 1000000.00,
    "cost": 600000.00,
    "quantity": 100,
    "ending_qty": 70
  },
  "classification": {
    "trade_type": "DOWN",
    "reason": "P001是S001的母公司"
  },
  "calculation": {
    "gross_margin_rate": 0.40,
    "unsold_ratio": 0.70,
    "tax_rate": 0.25,
    "unrealized_profit": 280000.00
  },
  "entries": [
    {"dr": "营业收入", "amount": 700000.00},
    {"cr": "营业成本", "amount": 420000.00},
    {"cr": "存货", "amount": 280000.00},
    {"dr": "递延所得税资产", "amount": 70000.00},
    {"cr": "所得税费用", "amount": 70000.00}
  ],
  "hash": "sha256:a3f8b2c19d4e7..."
}
```

### 5.2 哈希链

每笔分录的日志计算SHA256哈希值，并存入SQLite。日志的 `hash` 字段同时包含了上一笔分录的哈希，形成哈希链（类似区块链的简化版），任何篡改都会导致链条断裂。

```
ELIM-0001 ──hash──→ ELIM-0002 ──hash──→ ELIM-0003 ──hash──→ ...
```

### 5.3 未来向区块链的迁移路径

当前哈希链存储在本地SQLite中。若未来对接企业级区块链（如DRP系统的T33模块），迁移仅需：

1. 将 `log_hash` 字段的值通过T33的RESTful API发送到链上
2. 本地保存链上返回的 `tx_id`（交易哈希）
3. 追溯查询时，用 `tx_id` 在链上验真

**核心计算逻辑和日志结构无需任何修改。**
